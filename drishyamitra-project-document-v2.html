01 — Intent

Why this project exists

Most student portfolio projects are CRUD applications dressed up with a modern UI. Drishyamitra is different — a system where the engineering is the point, and every technology choice has a concrete justification.

"The most impressive thing in this project is not the face recognition. It is the agentic tool-calling layer — and most students building this will never get there."

The project targets three things placement interviewers at AI-focused companies actually probe: understanding LLMs as reasoning engines with tool-calling capabilities, the ability to design background processing systems, and engineering discipline in polyglot microservice architectures where each technology is justified by a real reason.

The core problem being solved

A person with 10,000 photos cannot find a specific photo without scrolling. They cannot search by face. They cannot say "send my grandmother's birthday photos to my mother" and have it happen. Traditional apps solve storage. Drishyamitra solves retrieval and action — the two harder problems. And when delivery hits platform size limits, it handles that gracefully too.

What a user experiences
Upload photos once. Faces detected automatically. Label a person once — the system remembers forever. Ask in plain language: "Show me Dad's photos from Diwali 2023." The AI finds them. Say "Email them to mom." The agent does it — and if the total size exceeds Gmail's limit, asks first before sending a compressed ZIP.
What the system actually does
DeepFace extracts 512-dim face embeddings. Groq agent resolves multi-step commands via tool calling. BullMQ workers process jobs without blocking the API. Delivery service checks payload size before sending — triggering a confirmation flow and archiver-based ZIP compression when limits are exceeded. Socket.io streams every event in real time.

What this proves to an interviewer

  • You understand LLMs as reasoning engines with tool-calling, not just text generators
  • You can design background processing systems with proper queue management and failure handling
  • You can build polyglot architectures where language choice per service is technically justified
  • You handle real-world delivery constraints (file size limits) with a proper UX flow, not a silent failure
  • You understand the tradeoffs between synchronous and asynchronous processing
  • You design for conversation state and reference resolution — not just single-turn queries

03 — New feature

Smart ZIP delivery

When a delivery request exceeds platform size limits, the system does not fail silently and does not send anyway. It detects the oversize condition, informs the user, and asks for confirmation before compressing and sending a ZIP.

⚑
Size limits

Gmail (Nodemailer SMTP): 25MB total attachment size. WhatsApp (whatsapp-web.js): 100MB per message. These are checked before any delivery is attempted using Cloudinary's resource.bytes metadata — no actual download required at check time.

The user experience

Implementation — server side

// backend/src/services/delivery.service.js

const SIZE_LIMITS = {
  email: 25 * 1024 * 1024,     // 25MB Gmail SMTP limit
  whatsapp: 100 * 1024 * 1024  // 100MB whatsapp-web.js limit
};

export async function checkDeliverySize(photoIds, medium) {
  const photos = await Photo.find({ _id: { $in: photoIds } })
    .select('cloudinaryPublicId bytes');

  // Use stored bytes field — no download needed
  const totalBytes = photos.reduce((sum, p) => sum + (p.bytes || 0), 0);

  return {
    totalBytes,
    exceedsLimit: totalBytes > SIZE_LIMITS[medium],
    limit: SIZE_LIMITS[medium],
    count: photos.length
  };
}

export async function createAndUploadZip(photoIds, userId) {
  const archive = archiver('zip', { zlib: { level: 6 } });
  const chunks = [];

  archive.on('data', chunk => chunks.push(chunk));

  for (const id of photoIds) {
    const photo = await Photo.findById(id);
    const stream = await downloadFromCloudinary(photo.imageUrl);
    archive.append(stream, { name: `${photo._id}.jpg` });
  }

  await archive.finalize();
  const buffer = Buffer.concat(chunks);

  // Upload ZIP to Cloudinary as raw file, auto-delete after 24hr
  const result = await cloudinary.uploader.upload_stream(
    { resource_type: 'raw', folder: 'zips',
      invalidate: true, tags: ['temp-zip'] },
    buffer
  );

  return result.secure_url;
}

Implementation — the confirmation loop

// Inside the sendEmail / sendWhatsApp tool executors

async function executeSendEmail({ recipient, photoIds }, session, io) {
  const ids = photoIds?.length ? photoIds
    : session.memory.lastPhotoSearch.photoIds; // reference resolution

  const { exceedsLimit, totalBytes, count } = await checkDeliverySize(ids, 'email');

  if (exceedsLimit) {
    // Store pending state in session — wait for user confirmation
    session.memory.pendingZipConfirmation = {
      medium: 'email', recipient, photoIds: ids,
      totalMB: (totalBytes / 1024 / 1024).toFixed(1)
    };

    // Push confirmation request to client via Socket.io
    io.to(session.userId).emit('delivery:zip-confirm', {
      medium: 'email', recipient, count,
      totalMB: session.memory.pendingZipConfirmation.totalMB,
      limitMB: 25
    });

    return {
      status: 'awaiting_confirmation',
      message: `Total size ${session.memory.pendingZipConfirmation.totalMB}MB exceeds Gmail's 25MB limit. Confirmation requested.`
    };
  }

  // Under limit — send normally
  return await sendEmailWithPhotos(recipient, ids);
}

// Separate endpoint — called when user confirms/declines in UI
// POST /api/delivery/zip-confirm { sessionId, confirmed: true|false }
export async function handleZipConfirmation(req, res) {
  const { sessionId, confirmed } = req.body;
  const session = await redis.get(`chat:${sessionId}`);
  const pending = session.memory.pendingZipConfirmation;

  if (!pending) return res.status(400).json({ error: 'No pending confirmation' });

  if (!confirmed) {
    delete session.memory.pendingZipConfirmation;
    await redis.setex(`chat:${sessionId}`, 86400, JSON.stringify(session));
    return res.json({ status: 'cancelled' });
  }

  // User confirmed — create ZIP and deliver
  const zipUrl = await createAndUploadZip(pending.photoIds, req.user.id);
  await deliverZip(pending.medium, pending.recipient, zipUrl, pending.photoIds.length);

  await DeliveryHistory.create({
    userId: req.user.id, photoIds: pending.photoIds,
    recipient: pending.recipient, medium: pending.medium,
    status: 'sent-zip', zipUrl
  });

  delete session.memory.pendingZipConfirmation;
  await redis.setex(`chat:${sessionId}`, 86400, JSON.stringify(session));

  req.io.to(req.user.id).emit('delivery:done', {
    medium: pending.medium, recipient: pending.recipient,
    count: pending.photoIds.length, format: 'zip'
  });

  res.json({ status: 'sent', zipUrl });
}

◈
Why this is architecturally clean

The size check uses Cloudinary's stored bytes metadata — no photo download happens at check time. The confirmation state lives in Redis session (ephemeral, correct) not MongoDB (permanent, wrong). The ZIP is uploaded to Cloudinary as a temporary raw asset and scheduled for deletion — no disk storage on the server. The entire flow is non-blocking; the BullMQ delivery worker handles the actual compression and upload.

⚑
One dependency to add

Install archiver in the backend: npm install archiver. This is the Node.js streaming ZIP library — well-maintained, zero native dependencies, streams directly to a buffer without writing to disk. No other changes to the package manifest are needed for this feature.

06 — Interview preparation

12 questions you will be asked

If this project is on your resume, an interviewer will probe at least four of these. Know all twelve before submitting applications.

Q1
How does the agent resolve "them" in "email them to my brother"?

Conversation history stored in Redis per session. Every Groq call receives full message history. Backend also maintains session.memory.lastPhotoSearch.photoIds — the sendEmail tool injects these directly rather than relying entirely on LLM reconstruction. Deterministic state augments probabilistic reasoning.
Q2
Why Redis for session memory instead of MongoDB?

Session memory is ephemeral (24hr TTL), updated on every turn, and requires sub-millisecond reads since it blocks the agent loop. MongoDB's persistence overhead and connection latency are unnecessary for short-lived session-scoped data.
Q3
Why BullMQ instead of synchronous processing?

Face recognition takes 2–10 seconds per photo. Synchronous processing blocks the Express event loop. BullMQ moves work to background workers, the API returns immediately, Socket.io streams progress. Industry-standard pattern for any long-running CPU task in Node.
Q4
Why a Python microservice for face recognition?

DeepFace with FaceNet512 and RetinaFace produces measurably better embeddings than JS alternatives in difficult lighting and off-angle conditions. Python is the correct tool for this problem. Node calls it over HTTP internally — the boundary is clean and the Node side never touches Python code.
Q5
How do face embeddings work and how do you match them?

FaceNet512 maps each face to a 512-dimensional float vector where geometrically similar faces cluster close together. Cosine similarity measures angular difference between vectors — preferred over Euclidean distance because it's invariant to vector magnitude, making it more robust across different image brightness conditions.
Q6
Why not Qdrant?

At a few thousand faces, cosine similarity in Node against MongoDB is under 20ms — fast enough. Qdrant's HNSW indexing becomes valuable at tens of thousands of vectors. The embedding schema is designed for clean migration — moving to Qdrant in v2 requires only a different query layer, no data changes. Adding it to v1 would be resume padding I cannot justify technically.
Q7
How do you handle tool-calling failures or hallucinations?

Parameters validated before execution — people coerced to array, recipients validated. Tool errors returned to Groq as tool results, not thrown. Agent loop bounded at 5 iterations. Users get friendly messages, never raw stack traces. pendingZipConfirmation auto-expires after 10 minutes to prevent zombie session state.
Q8
How does Node communicate with the Python service?

HTTP POST to Flask at an internal URL. Node polls GET /health on startup and refuses to queue face jobs if it returns an error. 3-attempt retry with 30s timeout on each recognition call. Failure is surfaced to the client via Socket.io, not silently dropped.
Q9 — new
How does the ZIP delivery feature work?

Before sending, the delivery service sums photo sizes using Cloudinary's stored bytes metadata — no download required. If total exceeds the platform limit (25MB for Gmail, 100MB for WhatsApp), a delivery:zip-confirm Socket.io event is emitted to the client. On confirmation, archiver streams photos into a ZIP buffer, uploads it to Cloudinary as a temporary raw asset, and sends the URL. The ZIP is auto-deleted after 24 hours. All outcomes logged to DeliveryHistory.
Q10 — new
Why Socket.io for the ZIP confirmation instead of a REST endpoint?

The confirmation is a server-initiated event — the server discovers the size issue during delivery execution and needs to notify the client immediately, not wait for the client to poll. Socket.io is already in the stack for recognition progress events. Reusing the existing connection for delivery confirmation is architecturally consistent and requires no additional infrastructure.
Q11 — new
Why was Socket.io added minimally in Sprint 1 instead of fully?

Sprint 2 (agent framework) is entirely HTTP-based and has no real-time use case. Adding full Socket.io in Sprint 1 would be building infrastructure for a problem that doesn't exist yet. The minimal Sprint 1 layer (JWT handshake + user room join only) solves the non-obvious auth setup early — the one part that causes debugging pain if discovered in Sprint 4 while simultaneously working on BullMQ workers.
Q12
How would you scale this to 10,000 simultaneous users?

Horizontal BullMQ workers for recognition queue load. Redis cluster for session memory. Move embeddings to Qdrant with HNSW indexing as face database grows past ~50k. Cloudinary CDN handles photo serving at any scale without changes. Python face service scaled horizontally behind a load balancer. Each deferred decision becomes justified precisely at the scale that demands it.
